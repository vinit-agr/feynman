# Transcript Session Management — Design Spec

## Overview

Add first-class project grouping and session management to the Claude transcripts view. Users can organize, move, and delete transcript sessions through a unified UI. All changes persist to Convex backend. The ingestion pipeline respects manual grouping and soft-delete state.

## Data Model

### New Table: `projects`

First-class entity representing a project group.

| Field | Type | Description |
|-------|------|-------------|
| `name` | `v.string()` | Display name (e.g., "feynman") |
| `source` | `v.string()` | Scopes projects per source (e.g., "claude-transcripts") |
| `order` | `v.optional(v.number())` | Manual sort position. Absent = sort by `lastActivityAt` |
| `lastActivityAt` | `v.number()` | Timestamp of most recent session. Updated on session add/move/delete. Used for default sort when `order` is absent |
| `createdAt` | `v.number()` | Creation timestamp |

**Indexes:** `by_source(source)`, `by_source_order(source, order)`, `by_source_name(source, name)`

### Changes to `rawFiles`

| Field | Type | Description |
|-------|------|-------------|
| `projectId` | `v.optional(v.id("projects"))` | Reference to the project this file belongs to |
| `deleted` | `v.optional(v.boolean())` | Soft-delete flag. `false` = active (default on creation). `true` = hidden from UI, skipped by ingestion. All new writes must set this explicitly. Queries filter with `deleted !== true` to handle both `false` and `undefined` (legacy rows) |
| `storageId` | `v.optional(v.id("_storage"))` | **Changed from required to optional.** Zero-message files have no stored content |

Existing fields `projectName` and `projectPath` remain as ingestion metadata. `projectId` is the source of truth for grouping.

**New indexes:** `by_projectId(projectId)`, `by_source_deleted(source, deleted)`

**Filtering convention:** All queries that return rawFiles to the UI must filter out rows where `deleted === true`. The `getBySourceId` query (used by ingestion) returns the row regardless of `deleted` status so the ingestion script can detect previously seen files.

### `knowledgeEntries` — No Changes

Entries link to rawFiles via `rawFileId`. Project relationship is derived: entry → rawFile → project.

**Note on delete behavior:** When a rawFile is soft-deleted, its knowledgeEntries are hard-deleted. These can be re-generated by re-extraction if the rawFile is later recovered. This is acceptable because extraction is a deterministic/repeatable operation (mechanical extractors) or a re-runnable operation (AI extractors).

## Backend Mutations & Queries

### Projects

- **`projects.create(name, source)`** — Create project, set `order` to max+1, set `lastActivityAt` to now
- **`projects.rename(projectId, newName)`** — Update name
- **`projects.delete(projectId)`** — Set `deleted: true` on all rawFiles in project, delete their knowledgeEntries, hard-delete the project row. Note: this is an intentional hard delete — user-initiated project deletion removes the project permanently. This is distinct from the cleanup script which preserves projects
- **`projects.reorder(projectIds[])`** — Accept ordered array of IDs, update `order` on each (0, 1, 2, ...)
- **`projects.listBySource(source)`** — Return all projects. Sort by `order` if set, then by `lastActivityAt` descending for unordered projects (unordered projects appear after manually ordered ones)
- **`projects.findOrCreate(name, source)`** — Used by ingestion. Find by name+source or create new. Returns projectId

### Raw Files

- **`rawFiles.moveToProject(rawFileId, projectId)`** — Update `projectId` on the rawFile. Update `lastActivityAt` on both old and new projects
- **`rawFiles.softDelete(rawFileId)`** — Set `deleted: true`, delete associated knowledgeEntries. Update `lastActivityAt` on the project
- **`rawFiles.listByProject(projectId)`** — Non-deleted rawFiles for a project, sorted by timestamp desc
- **`rawFiles.listUngrouped(source)`** — Non-deleted rawFiles where `projectId` is absent, sorted by timestamp desc. Implementation: query `by_source_deleted` index with `source` and `deleted === false`, then filter in code for absent `projectId`
- Existing `rawFiles.list` and `rawFiles.getBySourceId` — `list` filters out `deleted: true`; `getBySourceId` returns regardless of deleted status (for ingestion dedup)

## Frontend

### Unified Session View

Replaces the current two-tab ("Extracted Entries" / "Raw Files") layout with a single view.

**Main area — Project accordions:**
- Collapsible accordion per project, sessions listed within
- Drag handle on project headers for manual reordering
- Drag handle on session rows for moving between projects
- Project header shows: name, session count, Rename button, Delete button
- "+ New Project" button at top (creates empty project with inline name edit)
- "Ungrouped" section at bottom for sessions without a project (no rename/delete)
- Sessions sorted by timestamp (descending) within each project

**Session interactions:**
- **Click** session row → opens slide-over
- **Drag** session row → move to another project (auto-sorts by timestamp on drop)
- **Context menu (⋯)** → "Move to →" (submenu listing projects + "New Project...") and "Delete session"

**Confirmation dialogs:**
- Delete session: "Delete session '{title}'? The transcript will be hidden and extracted content will be removed. You can recover the transcript later, but extraction will need to be re-run."
- Delete project: "Delete project '{name}' and its {count} sessions? All transcripts will be hidden and extracted content will be removed. Transcripts can be recovered later, but extraction will need to be re-run."

### Unified Slide-Over

Replaces the separate EntrySlideOver and RawFileViewer with a single component.

**Header:**
- Session title (derived from first human message)
- Metadata: session ID (truncated), timestamp, message count

**View dropdown:**
- One option per extractor registered for the source (e.g., "Project Work Summary", "Engineering Decisions")
- "Raw Transcript" as the last option (always available)
- Default: first extractor

**Content area:**
- **Extractor views:** Rendered/Raw toggle showing the extracted knowledge entry content as markdown or plain text
- **Raw Transcript view:** Parsed/Raw toggle — Parsed shows JSONL records as collapsible list (role, preview, timestamp); Raw shows full text

## Ingestion Pipeline Changes

```
For each discovered JSONL file:
  1. Derive sourceId, projectName from path (unchanged)
  2. Check if rawFile exists by sourceId (getBySourceId returns regardless of deleted status)
     - If exists:
       - If deleted === true → skip entirely (no upload, no update)
       - If not deleted → compare size/mtime, re-upload if changed. Do NOT touch projectId.
     - If not exists:
       - Parse file, check message count
       - If zero messages → create rawFile row with deleted: true, no storageId, no extraction
       - If has messages →
         - findOrCreate project by (projectName, "claude-transcripts")
         - Upload file to storage
         - Create rawFile with projectId, storageId, deleted: false
         - Enqueue auto-run extractors
```

**Edge case — previously zero-message file gains messages:**
A file previously ingested as zero-message (`deleted: true`, no `storageId`) may later gain messages. On re-ingestion, the script sees `deleted === true` and skips it. This is correct — the file was marked as "seen and empty." If the user wants to re-ingest it, they can run `cleanup:claude` to reset all rawFiles and re-ingest from scratch.

**Mutation contract changes:**
- `rawFiles.create` args must accept `storageId` as optional (currently required)
- `rawFiles.create` args must accept `projectId` and `deleted`
- `rawFiles.reupload` remains unchanged (always has a `storageId`)
- Existing `rawFiles.countBySource` query must also exclude `deleted === true` rows

**Key invariants:**
- `storageId` and file storage are never modified by grouping/delete operations
- `sourceId` remains the ingestion dedup key (based on filename, not project)
- Manual `projectId` assignments are never overwritten by re-ingestion
- The skip/upload/update decision uses `sourceId`, `localFileSize`, `localModifiedAt` — not `projectName`
- `deleted` is always set explicitly to `true` or `false` on creation — never left absent

## Migration Strategy

Clean slate approach using a reusable cleanup script:

**`pnpm cleanup:claude` script (order matters):**
1. Collect all `storageId` values from `rawFiles` where `source = "claude-transcripts"` (before deleting the rows)
2. Delete all `knowledgeEntries` where `source = "claude-transcripts"`
3. Delete all `rawFiles` where `source = "claude-transcripts"`
4. Delete collected storage files from Convex file storage
5. **Do NOT delete `projects`** — projects remain as empty groups and are reused by `findOrCreate` during re-ingestion

Then run `pnpm ingest:claude` to re-ingest everything with the new schema. The cleanup script is reusable for future resets.

## Key Design Decisions

1. **Projects as a first-class table** (not derived from folder path) — enables rename, reorder, empty projects, and independent lifecycle
2. **Soft-delete via `deleted` boolean on rawFiles** — preserves sourceId for ingestion dedup, enables future recovery UI. knowledgeEntries are hard-deleted on session delete (re-extractable)
3. **`projectId` on rawFiles only** — knowledgeEntries derive project through rawFileId relationship, no duplication
4. **Ingestion respects manual grouping** — new files auto-grouped by folder path, existing files never have projectId overwritten
5. **Zero-message files** — rawFile row with `deleted: true` and no `storageId` acts as "seen and skipped" marker
6. **Unified view** — single accordion view with extractor dropdown in slide-over, replacing two-tab layout
7. **`storageId` optional** — allows zero-message file tracking without wasting storage. All code that accesses `storageId` must handle the absent case (only relevant for the Raw Transcript view in the slide-over, which should show "No content available" for zero-message files)
8. **`lastActivityAt` on projects** — denormalized timestamp for efficient default sorting without cross-table joins. Updated by mutations that add, move, or delete sessions
9. **Projects preserved during cleanup** — cleanup script keeps project rows so manual naming/ordering survives re-ingestion
